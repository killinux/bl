一、背景与动机

Blender 是开源 3D 创作套件，内置完整的 Python API（bpy），使得程序化控制成为可能。随着大语言模型的发展，业界开始探索用 AI 辅助 3D 工作流 —— 从自然语言生成场景到自动化渲染管线。

核心诉求包括：

  • 插件开发辅助 —— 让 AI 帮写 Blender 插件的样板代码和业务逻辑
  • 场景快速原型 —— 用自然语言描述场景，AI 生成对应的 bpy 代码
  • 批量自动化 —— 重命名、导出、渲染等重复操作
  • 调试排错 —— 把报错贴给 AI，快速定位问题

二、现有方案全景

1. blender-mcp 主流

github.com/ahujasid/blender-mcp · 2000+ Stars · MCP 协议

通过 MCP 协议连接 Claude Desktop / Cursor 等客户端与 Blender，支持场景查询、代码执行、资产库集成、AI 3D 生成。

Claude Desktop ── MCP ──> blender-mcp server ── Socket ──> Blender addon (:9876)

2. BlenderGPT 经典

github.com/gd3kr/BlenderGPT · 5000+ Stars · OpenAI API

Blender 内置面板，用户输入自然语言，直接调用 OpenAI GPT-4 API 生成 bpy 代码并执行。最早的 Blender+AI 项目之一，简单直接但仅支持 OpenAI。

3. Headless 模式 稳定

Blender 原生 · 无需额外组件

Claude Code 写 Python 脚本，通过 blender --background --python script.py 执行。不需要任何桥接，最稳定，适合 CI/CD 和批量任务。

# Claude Code 写好脚本后
blender --background scene.blend --python render_script.py -- --output /tmp/render.png

4. 自定义 REST/WebSocket 桥接 定制

各工作室自研

在 Blender 内运行 Flask/FastAPI 服务，暴露 REST API。灵活度高，可针对工作室需求定制，但需要自行开发维护。

5. 我们的 Remote Bridge 跨网络

本项目 · Relay 中转 + blender-mcp Socket

解决 Claude Code 在 AWS、Blender 在公司内网 Mac 的场景。Mac 主动轮询 AWS，通过 blender-mcp 插件的 Socket 接口控制 Blender。

Claude Code (AWS) ── HTTP ──> Relay Server (:8080) <── poll ── Bridge (Mac) ── TCP ──> blender-mcp (:9876)

三、blender-mcp 深度分析

3.1 API 能力（22 个工具）

分类数量核心 API
基础操作4execute_code、get_scene_info、get_object_info、get_viewport_screenshot
PolyHaven 资产库5搜索/下载免费 HDR、材质、模型
Sketchfab 模型库4搜索/预览/下载 3D 模型
Hyper3D Rodin5文字/图片生成 3D 模型（AI）
Hunyuan3D (腾讯)4文字/图片生成 3D 模型（AI）

关键发现：实际上核心就是 execute_code，其他所有操作（创建物体、改材质、灯光、渲染）都是通过它执行 bpy 代码。没有专门的高层 API。

3.2 通信协议

层协议说明
AI 客户端 ↔ MCP ServerMCP (stdio)标准 MCP 协议，通过 npx/uvx 启动
MCP Server ↔ Blender AddonTCP SocketJSON 消息，端口 9876，localhost

Socket 消息格式：

// 发送
{"type": "execute_code", "params": {"code": "import bpy; print(bpy.data.objects.keys())"}}

// 接收
{"executed": true, "result": "['Camera', 'Cube', 'Light']\n"}

3.3 优势与局限

优势

设置简单，一条命令安装
MCP 原生集成，自然语言交互
资产库集成（PolyHaven、Sketchfab）
AI 3D 生成（Hyper3D、Hunyuan3D）
社区活跃，2000+ Stars

局限

必须同一台机器（localhost Socket）
截图需传 filepath，不直接返回图片
无 undo 集成，出错需手动清理
复杂场景需大量来回交互
AI 无空间推理能力，不知道物体实际位置关系

五、主流工作流

5.1 插件/脚本开发（最实用）

推荐指数：★★★★★

Claude Code 写 Blender 插件代码（bl_info、Operator、Panel），用户安装到 Blender 里测试，迭代修改。这是最可靠的工作流，因为 Claude 写的是文件而非实时执行。

# 典型流程
1. 告诉 Claude 插件需求
2. Claude 写出 .py 插件文件
3. 推送到 Mac（cli.py push）
4. Blender 里安装并测试
5. 截图看效果 → 反馈 → 迭代

5.2 场景快速原型

推荐指数：★★★★★

用自然语言描述场景，AI 生成 bpy 代码创建物体、材质、灯光。适合打草稿和概念验证，不适合最终生产级模型。

"创建一个室内场景：一个房间，中间放一张桌子，四把椅子围着桌子，天花板上一盏暖色吊灯"

5.3 批量自动化

推荐指数：★★★★★

重命名所有物体、批量应用修改器、统一导出格式、批量渲染等。AI 生成脚本后一次性执行，效率极高。

5.4 材质/Shader 开发

推荐指数：★★★★★

用代码创建节点树极其冗长（每个节点要设位置、参数、连线），简单材质可以，复杂节点树手动搭更快。

5.5 调试排错

推荐指数：★★★★★

把 Blender 控制台报错贴给 Claude，快速定位问题原因。也可以让 Claude 分析场景状态（物体重叠、缺失材质等）。

5.6 渲染管线

推荐指数：★★★★★

配置 Cycles/EEVEE 参数、设置渲染层、管理输出路径。适合 headless 模式批量渲染。

六、最佳实践

6.1 小步迭代

一次让 AI 做一件事，完成后截图确认，再做下一步。不要一句话要求整个复杂场景。

# 好的做法
"先创建一个立方体" → 截图 → "把它变成桌子形状" → 截图 → "加一个木纹材质"

# 不好的做法
"创建一个完整的客厅场景，包含沙发、茶几、电视柜、地毯、台灯、窗帘和室内光照"

6.2 截图驱动开发

每次操作后截图，让 Claude 看到效果。这是我们 Bridge 方案的核心优势，弥补了 AI 缺乏空间推理的短板。

做 → 截图 → 看 → 改 → 截图 → 确认

6.3 版本声明

告诉 Claude 你用的 Blender 版本。3.x 和 4.x 的 API 有显著差异：

  • 4.x 使用 bpy.context.temp_override()，3.x 使用 context.copy()
  • 部分操作符参数有变化
  • 节点树 API 有细微调整

6.4 先审后跑

执行前审查生成的代码，尤其是涉及删除、修改现有物体的操作。blender-mcp 没有 undo 集成。

6.5 利用 Headless 模式

对于不需要实时交互的任务（批量渲染、文件转换），直接用 blender --background --python 更稳定高效。

6.6 安装类型提示

安装 fake-bpy-module 让 Claude 写代码时有更好的 API 感知：

pip install fake-bpy-module-3.6

七、常见陷阱

陷阱说明建议
空间推理 AI 不知道物体是否重叠、悬浮、穿模 每步截图确认，手动调整位置
精细建模 顶点级操作靠自然语言不靠谱 用 Blender 原生工具手动建模
版本差异 AI 可能生成旧版 API 调用 在 CLAUDE.md 中声明 Blender 版本
节点坐标 AI 创建的节点树节点位置重叠 手动加 node.location = (x, y)
无 Undo MCP 执行的操作无法 Ctrl+Z 操作前保存 .blend 文件
大场景性能 get_scene_info 在大场景下数据量大 blender-mcp 默认限制 10 个物体
UV/绑骨/物理 需要视觉反馈和反复调试 手动操作，AI 仅辅助写脚本

八、我们的方案定位

8.1 解决了什么问题

行业现有方案（blender-mcp、BlenderGPT）都假设 AI 客户端和 Blender 在同一台机器上。我们的场景是：

  • Claude Code 运行在 AWS 云服务器
  • Blender 运行在公司内网 Mac，无公网 IP，防火墙禁止入站连接
  • Mac 可以发出 HTTP 请求（可以上网）

通过 Relay 中转 + Mac 主动轮询 的"拉模式"，绕过防火墙限制。

8.2 相比行业方案的差异化优势

1. 跨网络

唯一支持 AI 和 Blender 在不同网络的方案。不需要 VPN、SSH 隧道、Tailscale 等工具。

2. 视觉反馈回路

截图传回 AWS，Claude 能"看到"场景状态。blender-mcp 原生虽有截图 API 但只面向 Claude Desktop，我们将其扩展到了 Claude Code CLI。

3. 复用 blender-mcp 生态

不重复造轮子，直接适配 blender-mcp 的 Socket 协议，继承其全部 22 个 API 能力。

4. MCP Server 封装

将远程 Bridge 封装为 MCP Server，Claude Code 可以像本地 blender-mcp 一样用自然语言控制。

九、未来演进方向

1. Headless 渲染回传

不只截取视口，还触发 Cycles/EEVEE 渲染，将渲染结果传回 AWS。实现"描述场景 → 渲染 → 查看效果"的完整闭环。

2. 操作前后对比

每次执行操作前后各截一张图，Claude 对比确认效果。形成自动化的质量检查。

3. .blend 文件管理

支持从 AWS 推送 .blend 文件到 Mac 打开，或者从 Mac 回传 .blend 文件。实现完整的文件同步。

4. 多用户支持

Relay Server 支持多个 Bridge 连接，每个对应不同的 Blender 实例。适合团队协作场景。

5. WebSocket 替代轮询

如果防火墙允许 WebSocket（通常走 443 端口），可以将轮询改为 WebSocket 长连接，降低延迟。

十、结论

行业共识

  1. AI 最适合做 Blender 的"脚本助手"，而非"3D 艺术家"
  2. 插件开发、批量自动化、调试排错是 AI 最有价值的应用场景
  3. 场景原型可以用 AI 打草稿，但不能期望生产级质量
  4. 视觉反馈是关键 —— 没有截图/渲染回路的方案实用性大打折扣

我们方案的价值

在 Claude Code + 远程 Blender 的特定场景下，我们的 Bridge 方案是目前唯一可行的解决路径。它复用了 blender-mcp 的成熟生态，同时解决了跨网络和视觉反馈两个核心问题。
